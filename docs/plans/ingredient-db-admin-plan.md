# Flesh out the Ingredient DB Admin Page

## Context

The admin console (in the **fodmap-chat** frontend) ships with a disabled **"Ingredient DB / Soon"** sidebar button — the only admin section not yet built. Meanwhile FODMAP ingredient data isn't really a "database": it's a static Go map `data.FodmapDB` ([data/fodmap.go](../../data/fodmap.go)) compiled into the binary and **unconditionally** re-embedded into the vector search backend on every boot ([server/server.go:124](../../server/server.go)). There is no way to view, add, edit, or delete ingredients without a code change + redeploy.

**Goal:** make the Ingredient DB page a real **full-CRUD** admin tool — browse/search/filter all ingredients, create/edit/delete with persistence that survives restarts, view **level/group stats**, and **test the semantic search** (free-text → matched ingredient + certainty). This requires turning the static map into a persisted, editable catalog.

### Symbol-level claims verified with gopls
Run gopls/go with `GOFLAGS=-mod=mod` in this repo — the checked-in `vendor/` is out of sync with `go.mod` (inconsistent vendoring), so default (vendor) mode fails for both gopls and `go build`/`go list`. **This will also break `make check` until `go mod vendor` is run** (a write — deferred to implementation).
- `Searcher` interface implementations (`gopls implementation`): 3 production backends — [search/postgres.go:254](../../search/postgres.go), [search/weaviate.go:817](../../search/weaviate.go), [search/pinecone.go:186](../../search/pinecone.go) — **plus 3 test stubs** (chat_handler_test.go:151, direct_fodmap_client_test.go:24, handlers_test.go:39). The 3 stubs are why we add a narrow `FodmapWriter` rather than widening `Searcher`.
- `BatchUpsertFodmap` (`gopls references`): only one **production** caller — [server/server.go:124](../../server/server.go) (startup); all others are `_test.go`.
- `data.FodmapDB` (`gopls references`): consumed in production only at server/server.go:124. Static map feeds nothing but the startup upsert.

### Key architectural constraints (verified)
- **Postgres is always available** (`postgres-dsn` is required at startup for auth — [cli/serve.go](../../cli/serve.go)). The *vector search* backend is independently selectable (Postgres pgvector / Weaviate / Pinecone), and `s.searcher` may even be `nil`.
- The `Searcher` interface ([server/server.go:20-28](../../server/server.go)) only does single-result `SearchFodmap` + batch upsert — no list/paginate/filter, which is awkward on Weaviate/Pinecone.
- No ingredient name in `data.FodmapDB` contains `/` (verified) — so names are safe as URL path params (with encoding).
- Repo already uses the "type-assert to a narrower capability interface" idiom (`srv.Searcher().(menutracking.VectorSink)`).

## Approach: canonical Postgres catalog + vector index as derived projection

**Source of truth = a new relational `fodmap_catalog` table in the always-present auth Postgres DB.** Admin reads (list/get/stats) hit canonical only — relational gives `ILIKE`, `= ANY(groups)`, `OFFSET/LIMIT`, `COUNT GROUP BY` for free, on any deployment. The vector index is a one-way **derived projection** of the catalog, kept in sync only so `SearchFodmap` returns fresh metadata. This decouples CRUD correctness from whichever vector backend is configured.

### Backend (`fodmap-detector`)

0. **Fix vendoring first** — run `go mod vendor` (and `go mod tidy` if needed) so `vendor/modules.txt` matches `go.mod`. Without this, `go build`, `make check`, and gopls all fail in default mode. This is a prerequisite for everything below.

1. **`data/constants.go`** (MODIFY) — add exported `ValidFodmapLevels` (`high/moderate/low`) and `ValidFodmapGroups` (`fructans, GOS, lactose, excess fructose, sorbitol, mannitol`) slices for shared, server-authoritative validation.

2. **New canonical store package `fodmap/store/`** following the repo SQL convention (SQL in `sql/*.sql`, embedded via `//go:embed`, DDL in `schema.sql`, params `$1/$2` — never `fmt.Sprintf`; CLAUDE.md):
   - `sql/schema.sql` + `fodmap_meta(key TEXT PRIMARY KEY, value TEXT)` (the seed marker from step 4 — must be in the same DDL or seeding fails every boot). **Recommended default: `CREATE TABLE IF NOT EXISTS fodmap_catalog(ingredient TEXT PRIMARY KEY, level TEXT, groups TEXT[], notes TEXT, substitutions TEXT[], updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP)` with the app normalizing `ingredient` to lowercase on every write** (matching the repo's `TIMESTAMP DEFAULT CURRENT_TIMESTAMP` idiom at [auth/postgres_store.go:82-83](../../auth/postgres_store.go); `Update` should set `updated_at = CURRENT_TIMESTAMP` explicitly). All existing `data.FodmapDB` keys are already lowercase and *every* write goes through the handler, so app-level lowercasing fully enforces case-insensitive uniqueness via the plain PK — with **no new extension dependency** (the repo currently uses only the `vector` extension; `CITEXT` would add ops surface + a `CREATE EXTENSION citext` privilege requirement). `CITEXT PRIMARY KEY` remains a valid alternative if preserving original case in storage matters, but it's not recommended for v1.
   - `sql/{create,get,list,count,stats,update,delete,list_all}.sql`. `list.sql` builds its dynamic WHERE via the existing `text/template` + `renderSQL` pattern in [search/postgres.go:19-37](../../search/postgres.go) (skeleton templated, **values stay parameterized**). The `search` filter matches against both `ingredient` and `notes` so admins can find by concept, not just exact name.
   - `postgres.go` — `FodmapCatalogStore` with embedded SQL consts and methods: `EnsureSchema`, `Create` (INSERT, conflict → `ErrIngredientExists`), `Get` (`sql.ErrNoRows` → nil), `List(offset, limit, filter{search, level, group})`, `Count` (same filter struct as `List`), `Stats` (counts by level + by group), `Update` (rows-affected 0 → `ErrIngredientNotFound`), `Delete`, `ListAll` (for boot reload + seeding), `CountForSeed`. Define sentinels `ErrIngredientNotFound` / `ErrIngredientExists` (handlers branch with `errors.Is`). Doc comments on all exported identifiers.

3. **Narrow capability interface `FodmapWriter { UpsertFodmapItem(ctx, name, data.FodmapEntry) error; DeleteFodmapItem(ctx, name) error }`** (server package). Implement single-item upsert/delete in [search/postgres.go](../../search/postgres.go), [search/weaviate.go](../../search/weaviate.go), [search/pinecone.go](../../search/pinecone.go), reusing existing embed/upsert/delete code paths. **Do NOT widen the `Searcher` interface** — the admin handler type-asserts `s.searcher.(FodmapWriter)` (mirrors `menutracking.VectorSink`), so the existing Searcher test stubs in `server/*_test.go` stay untouched.

4. **Inject the catalog store as an interface (mirror `UserStore`), do NOT dial Postgres inside `server.New`.** This is a verified hard constraint: `New(ctx, cfg)` is production, but `NewServer(searcher, port)` and `NewServerWithChat(...)` are **test constructors that build a `*Server` with no Postgres** (they set `userStore: newMockStore()` directly — [server/server.go:178,199](../../server/server.go)). `UserStore` is an interface field on `Config` assigned at server.go:92. The catalog must follow the same pattern, or every `NewServer`-based test will try to hit a real DB.
   - Define a `CatalogStore` **interface** in the server package (the methods from step 2). Add a `CatalogStore` field to `Config` and a `catalogStore` field to `Server`; assign `cfg.CatalogStore` in `New`. **The test constructors `NewServer`/`NewServerWithChat` must also assign a stub `catalogStore`** (like `newMockStore()`), or admin-ingredient handlers nil-panic.
   - Construct the concrete `*FodmapCatalogStore` in **`cli/serve.go`** from the mandatory `postgresDSN` (step 6), `EnsureSchema`, pass via `Config.CatalogStore`. Separate `*sql.DB` pool (decision A below).

   **Startup seed/reload** (the highest-risk behavioral change — today server.go:124 clobbers edits): run inside `New`, guarded by `if cfg.CatalogStore != nil`, via a standalone testable `seedAndReload(ctx, catalog, searcher)`:
   - **Seed-once (NOT seed-if-empty)**: gate seeding on a persistent marker, not `COUNT==0`. A `COUNT==0` gate has a resurrection bug — an admin who deletes every ingredient and restarts gets the whole static map back (and combined with the prune gap, ghosts reappear). Use a tiny `fodmap_meta(key TEXT PRIMARY KEY, value TEXT)` row (`seeded=true`); seed `data.FodmapDB` only if the marker is absent, in one transaction with `ON CONFLICT (ingredient) DO NOTHING` + set the marker (neutralizes concurrent-boot race).
   - If `s.searcher != nil`: `EnsureFodmapSchema`, then `BatchUpsertFodmap` fed from **`catalog.ListAll()`** (not the static map) — this is what makes edits survive restarts. Keep the existing non-fatal `slog.Warn`-on-failure so a vector hiccup doesn't block boot.

5. **`server/admin_ingredients_handler.go`** (NEW) — handlers registered under the existing `adminMid` (jwtAuth → adminRequired) in [server/server.go:241-253](../../server/server.go):
   - `GET /api/v1/admin/ingredients` — list; query filters `search`, `level`, `group`, `page`, `limit` (optional filters as query params per CLAUDE.md).
   - `GET /api/v1/admin/ingredients/stats` — counts by level + group.
   - `GET /api/v1/admin/ingredients/search-test?q=` — calls `SearchFodmap`, returns match + certainty (503 if `searcher == nil`).
   - `GET /api/v1/admin/ingredients/{name}` — get one.
   - `POST /api/v1/admin/ingredients` — create (409 on conflict). Server normalizes `name` to trimmed form; rejects empty/whitespace-only names and names longer than 200 characters with 400.
   - `PUT /api/v1/admin/ingredients/{name}` — **strict update only** (404 if absent; **name is immutable** — no rename in v1). Does not upsert; admins must use `POST` to create.
   - `DELETE /api/v1/admin/ingredients/{name}`.
   - Go 1.22 mux resolves literal `/stats` & `/search-test` over `{name}` automatically (a guard test documents this).
   - **Validation** (server-authoritative): non-empty trimmed name (max 200 chars, no control characters), `level ∈ ValidFodmapLevels`, every group ∈ `ValidFodmapGroups` and deduplicated → else 400. Empty `groups`/`substitutions` are valid. Names are stored with leading/trailing whitespace removed; case is preserved in display but used for case-insensitive uniqueness.
   - **Sync policy (eventual consistency — no cross-store txn possible):** canonical write commits first and is authoritative. Then best-effort `FodmapWriter` sync; on sync failure return success with a `"warning":"search index sync pending"` field + `slog.Error`. Next boot's reload-from-canonical self-heals. If `searcher` doesn't implement `FodmapWriter` or is nil, CRUD still fully succeeds against canonical.

6. **`cli/serve.go`** (MODIFY) — construct the concrete `*FodmapCatalogStore` from the mandatory `postgresDSN`, run `EnsureSchema`, assign to `Config.CatalogStore`, `defer Close()`. This is the only place the catalog store is wired to a real DB (mirrors how `userStore` is built here and injected).

## Known trade-offs (deliberate, documented)
- **Restart prune gap:** `BatchUpsertFodmap` only upserts, never prunes — a vector row for an ingredient deleted in the catalog survives until... it doesn't get re-added, but isn't actively removed either. v1 accepts this minor staleness (delete also calls `DeleteFodmapItem` so it's removed at delete time; the gap only matters if that sync failed). Active reconcile-delete on boot is a future item.
- **Dual storage when `postgres-search` is active:** `fodmap_catalog` (metadata, auth concern) and `fodmap_ingredients` (embeddings, search concern) co-reside in the same DB. Distinct names; catalog is authoritative.
- **Rename** is out of scope for v1 (name is the PK / immutable in edit). Because the PK is the display name, adding rename later requires a migration to a surrogate key + slug; accept that limitation for v1.
- **No audit trail / soft delete:** admin edits are hard updates/deletes with no `created_by`, `updated_by`, or `deleted_at`. v1 accepts this; add if compliance/ops requirements arise.
- **DDL privileges at runtime:** `EnsureSchema` runs on every boot, so the application DB user needs `CREATE TABLE` (and `CREATE EXTENSION` if using `CITEXT`) in production. Provide this capability or a separate `migrate` command; document the requirement.

## Risks & gaps (second-pass review)
Resolved in the design above: **seeding resurrection** (now seed-once via marker), **minimal UI kit** (hand-rolled modal + native select + toggle buttons), **startup testability** (extract `seedAndReload`). Remaining items to handle during implementation:

- **Create-conflict detection (correctness):** the catalog uses `database/sql` + `sql.Open("pgx", dsn)` + `lib/pq` arrays (matching [search/postgres.go](../../search/postgres.go) and [auth/postgres_store.go](../../auth/postgres_store.go)). `Create` must map a Postgres unique-violation (**SQLSTATE `23505`**) on the `ingredient` PK (which is now case-insensitive) to `ErrIngredientExists` → 409. A bare INSERT error otherwise surfaces as a 500.
- **TEXT[] scanning:** the existing `pgxStringArray` scan helper in search/postgres.go is unexported. The new package needs its own helper (or use `pq.Array(&dest)` as the `sql.Scanner`) — the pgx stdlib driver doesn't scan arrays natively, which is why search wrote that helper. Replicate it.
- **`search-test` no-match handling:** `SearchFodmap` returns a **non-sentinel** `fmt.Errorf("not found")` *only* when the table is empty; otherwise pgvector always returns the nearest neighbor (even nonsense queries → low certainty). The handler must return `200 {match:null}` (not 500) on that error path, and the UI should frame results as "closest match" + show the certainty score. `searcher == nil` → 503.
- **`search-test` certainty threshold:** define a threshold below which the UI labels the result "low confidence / may not match". v1 uses 0.6 as a visual guideline; the handler still returns the raw score so callers can decide.
- **Stats group semantics:** ingredients carry multiple groups, so group counts **sum to more than the total** — present as a tag distribution, not a partition, in both API and UI. Return `total_count` alongside level/group distributions so the UI does not have to infer totals.
- **Pagination totals:** `Count` must accept the **same filter struct** as `List`, or page counts drift when filters are active.
- **DB connections:** catalog opens its own `*sql.DB` pool to the shared auth DSN (a second pool alongside `userStore`); acceptable, or thread an existing `*sql.DB` through `Config`. Pick one explicitly.
- **Validation niceties (server-side):** trim name (max 200 chars, reject control characters), dedup groups, reject unknown/duplicate groups and empty level; these are correctness, not just UX.
- **Seed-once transaction ordering:** the marker insert must be part of the same transaction as the seed inserts (or inserted atomically first with `ON CONFLICT DO NOTHING`) so a crash mid-seed cannot leave the marker absent while rows are present, causing partial re-seed on next boot.
- **URL-encoding round-trips:** names may contain `&`, `?`, `#`, `%`, spaces, and hyphens. Handler tests must verify `100% fruit juice` and `salt & pepper` round-trip correctly through `encodeURIComponent` → mux → `PathValue`.
- **Error response envelope consistency:** reuse the existing JSON error envelope from `server/handlers.go` (`{"error":"..."}`) for 400/404/409/503; the warning field is separate and additive on 200/201 responses only.
- **Per-backend `FodmapWriter` behavior:** define `UpsertFodmapItem` to re-embed and upsert a single item, returning errors only on transport/embedding failures; `DeleteFodmapItem` returns error only on transport failures, not on "not found". Add per-backend unit tests for these new methods in addition to handler tests.
- **Boot reload observability:** log the count and duration of the catalog-to-vector reload on every boot so regressions are visible as the catalog grows.
- **`fodmap_meta` DDL in schema.sql:** addressed in step 2 — `fodmap_meta` is now listed alongside `fodmap_catalog` in `schema.sql`. (Flagged here originally because an earlier draft created it only in step 4's prose, which would have made `EnsureSchema` skip it and break seeding every boot.)
- **Weaviate embedding prefix:** [search/weaviate.go](../../search/weaviate.go) line 782 embeds ingredient names with `"search_document: " + name`, while Postgres and Pinecone embed just the raw name. `UpsertFodmapItem` implementations must preserve this backend-specific prefix — getting it wrong silently breaks semantic search on Weaviate.
- **Per-backend ID schemes for `DeleteFodmapItem`:** no `DeleteFodmap` method exists in any backend today — all three only have `BatchUpsertFodmap`. The new `DeleteFodmapItem` implementations must use each backend's existing ID convention: Weaviate uses `uuid.NewSHA1(uuid.NameSpaceOID, []byte("fodmap_"+name))`, Pinecone uses `"fodmap-"+name`, Postgres uses `DELETE FROM fodmap_ingredients WHERE ingredient = $1`. These aren't guessable — they must replicate the exact ID format each backend already uses.
- **Pinecone batch-vs-single embedding difference:** Pinecone's `BatchUpsertFodmap` calls `c.embedder.EmbedBatch` for efficiency ([search/pinecone.go](../../search/pinecone.go) line 282), while Postgres and Weaviate use `EmbedSingle`. `UpsertFodmapItem` will use `EmbedSingle` for all backends. This is a minor behavioral difference — per-item embeddings may differ slightly from batch-embedded ones in some models. Acceptable for v1, but worth documenting.
- **`FodmapWriter` package placement:** the existing analogous interface `menutracking.VectorSink` lives in the `menutracking` domain package, not `server`. The plan puts `FodmapWriter` in the `server` package, which is functional but creates a slight inconsistency with the established pattern. Consider placing it alongside `Searcher` or in a `fodmap` package if one emerges.
- **`Server` struct field:** the plan says "pass into `server.Config`" (step 6) but doesn't explicitly call out that the `Server` struct in `server/server.go` also needs a new `catalogStore` field (and that `New` must assign it). Easy to miss during implementation.
- **Mixed SQL embedding patterns in `fodmap/store/`:** the new package needs both static `//go:embed sql/*.sql` (for simple queries like `create`, `get`, `delete`) **and** the `text/template` + `renderSQL` machinery (for dynamic `list.sql` WHERE construction). This mirrors a hybrid of the `auth/postgres_store.go` pattern (static embeds) and the `search/postgres.go` pattern (templated SQL). The `renderSQL` function and `sqlParams` struct from [search/postgres.go:19-37](../../search/postgres.go) must be replicated in the new package — they can't be imported because they're unexported.
- **DB connection pool decision:** step 2 and step 6 imply a new `*sql.DB` pool for the catalog alongside the existing `userStore` pool. Two pools to the same DSN means double the connection overhead. Explicitly choose one: (A) a separate pool (simpler, matches the `search/postgres.go` pattern) or (B) thread the existing `userStore`'s `*sql.DB` through `Config` (more efficient, fewer connections). **Decision: use a separate pool (option A)** — it follows the established pattern in `search/postgres.go` and `auth/postgres_store.go`, keeps the catalog store self-contained, and the overhead is negligible for an admin tool.
- **`Config.PostgresDSN` comment is misleading:** the field comment says "required if PostgresSearch is true" but `cli/serve.go` makes it unconditionally required, and now the catalog store also requires it unconditionally. Update the comment to reflect reality.
- **Ingredient count:** `data.FodmapDB` contains 185 entries, not ~200. The verification step (step 3) should say "~185 ingredients".
- **Frontend asset reuse:** the admin ingredient page should reuse `FodmapResult` from `src/types/api.ts` and the `levelConfig`/`groupColors` from `IngredientSearch.tsx` for rendering stat cards and search-test results. Avoid redefining these in the admin page.
- **Accepted for v1 (document, don't fix):** no optimistic locking (last-write-wins on concurrent edits); name immutable (no rename); restart prune gap (orphan vector rows from a failed delete-sync persist until manually resynced). The public `GET /api/v1/search/fodmap/{ingredient...}` path reads the **synced** vector index, so it reflects admin edits — no stale-list risk there (verified: no public list-from-static-map endpoint exists).
